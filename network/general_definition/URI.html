URI

英特网中的每个英特网资源都有一个名字用于标识一个资源，资源名被称为统一资源标识符(Uniform Resource Identifier,URI)。它是一个标准的定义，并不属于某个协议的内容，很多人认为它是HTTP特有的，这是错误的。事实上，它是由因特网组织独立出来的内容，用于标识英特网资源，因此它不仅仅可以用于标识HTTP资源还可以用于其他英特网资源，比如FTP等等。URI有两种形式，分别为URL和URN，如下URI采用URL来标识一个HTTP图片资源:

http://www.joes-hardware.com/specials/saw-blade.gif

 

URL

统一资源定位符(URL)是资源标识符最常见的形式。它描述了一台特定服务器上某个资源的特定位置。现在，几乎所有的URI采用URL形式，下面是几个个URL实例:

 

http://www.joes-hardware.com/in?item=12731  ---> http资源

mailto:president@whitehouse.gov --->邮件资源

ftp://ftp.lots-o-books.com/pub/complete-price-list.xls  --->ftp资源

 

它包含三部分:

 

l 方案(scheme)部分，说明了访问资源所使用的协议类型，用来表明是何种资源这部分通常是HTTP协议(http://)，因特网客户端(通常是游览器)会根据scheme来确定使用哪个端口作为默认端口并使用何种访问协议

l 服务器的因特网地址部分，可以是IP或域名(比如,www.joes-hardware.com)

l 路径部分，指定了该资源在服务器中的路径(比如,/specials/saw-blade.gif)

 

但是，并非所有的URL都如上面一样是这三部分组成，这是因为英特网资源可以通过不同方案来访问的，因此URL语法会随方案的不同而有所不同

 

URL语法

资源是可以通过不同的方案（比如HTTP、FTP、SMTP)来访问的，因此URL语法会随方案的不同而有所不同。但这并不意味着不同的URL方案会有完全不同的语法，大部分URL都遵循通用的URL语法，而且不同的URL方案风格和语法都有不少重叠，另外，URL不区分大小写

大多数的URL方案的URL语法都建立在这个由9部分构成的通用格式上:

<scheme>://<user>:<password>@<host>:<port>/<path>;<params>?<query>#<frag>

1. scheme

方案指定访问资源使用什么协议，方案组件必须以一个字母符号开始，由’:’将其与URL其余部分分割开来

2. user与password

很多服务器提供某些资源都要求输入用户名和密码才会允许用户访问数据。FTP服务器就是这样一个常见的例子。例如:通过游览器访问ftp资源时，如果没有在URL中指定则游览器会弹出用户和密码框要求用户输入用户名和密码

3. host与port

主机与端口，要想在因特网上找到资源，应用程序要知道是那台机器装载了资源，以及在那台机器的什么地方可以找到能对目标资源进行访问的服务器。URL的主机和端口组件提供了这些信息

主机组件标识了宿主服务器，可用主机名(域名),或者IP地址，端口标识了服务器提供该资源的网络端口

4. path

路径说明了资源位于服务器的什么地方。路径通常很像一个分级的文件系统路径。比如:

http://www.baidu.com:80/a/b.html

这个URL中的路径为/a/b.html，很像UNIX文件系统中的文件系统路径。路径是服务器定位资源时需要的信息

如上，URL路径可用’/’字符将路径组件划分成为一些路径段，每个路径段都有自己的参数(param)组件

5. params

对于很多方案来说，只有简单的主机名和到达对象的路径时不够的，参数用于对其定位信息的补充。比如，FTP有两种传输模式，二进制和文本形式，你肯定不希望以文本形式来传送二进制图片，这样的话可能会变得一团糟，因此，为了向应用程序提供它们所需要的输入参数，以便正确地与服务器进行交互，URL中的params组件就起到作用了

这个组件就是URL中的名值对列表，由字符’;’将其与URL其余部分(以及各名值对)分割开来。比如:

 

ftp://prep.ai.mit.edu/pub/gnu;type=d

如前所述，HTTP URL的路径组件可以分成若干路径段，每段都可以有自己的参数，比如:

http://www.hello.com/hammers;sale=false/index.html;graphics=true

6. query

查询字符串是为了这样的目的:对于很多资源，比如数据库服务，都是可以通过查询来缩小请求资源类型的范围。用’?’号来与其他URL组件分割，以一系列名值对的形式出现，名值对直接用’&’分割,如下:

http://www.baidu.com/index.html?item=1273&color=blue&size=large

7. frag

有些资源，比如HTML，除了资源级之外，还可以做进一步的划分。比如，一个带有章节的文档，资源的URL会指向整个文档，但是理想的情况是，能够指定资源中的某个章节

为了引用这部分资源或资源的一个片段，URL支持使用片段(frag)组件来表示一个资源内部的片段，片段用’#’号与其他URL组件分割。比如:

http://www.hello.com/tools.html#drills

相对URL

URL有两种方式：绝对的和相对的。到目前为止，我们只见过绝对URL。绝对URL中包含有访问资源所需的全部信息

另一方面，相对URL是不完整的。要从相对URL中获取访问资源所需的全部信息， 就必须相对于另一个，被称为其基础(base)的URL进行解析

相对URL是URL的一种便捷缩略记法。如果你手工写过HTML的话，可能就会发现相对URL是多么便捷了。如下是一个嵌入了相对URL的HTML文档实例:

<HTML>

<HEADxTITLE>Joe ' s Tools</TITLEx/HEAD>

<BODY>

<H1> Tools Page </Hl>

<H2> Hammers <H2>

<P> Joe's Hardware Online has the largest selection of <A HREF="./ hammers.html">hammers

</A> on earth.

</BODY>

</HTML>

加入该HTML文档是资源：http://www.joes-hardware.com/tools.html的HTML文档

在这个HTML文档中有一个包含了 URL ./hammers.html的超链接。这个URL看起 来是不完整的，但实际上是个合法的相对URL.可以相对于它所在文档的URL对 其进行解释，在这个例子中，就是相对于www.joes-hardware.com服务器资源/tools

使用缩略形式的相对URL语法，HTML的编写者就可以省略URL中的方案、主机 和其他一些组件了。这些组件可以从它们所属资源的基础URL中推导出来。其他资 源的URL也可以用这种缩略形式来表示，因此，上例基础URL为:http://www.joes-hardware.com/tools.html

用这个URL作为基础，可以推导出缺失的信息。我们知道资源名为./hammers.html, 但并不知道方案或主机名是什么。通过这个基础URL,可以推导出方案为http,主机为www.joes-hardware.com

URL字符

URL被设计之初要求能统一命名因特网上所有资源，这也意味着要能在不同协议中传送不同资源。然而这些协议在传输数据会使用不同机制，比如电子邮件协议所使用的传输方法就会剥夺一些特定字符，因此要使得URL能在各种协议之间传输，则URL只能使用相对较小范围的可用字符以满足在各个协议都能传输URL

URL被设计之初，设计者的一个目标就是使得URL简单易读，因此设计者把URL设计为可读的。因此，不可见、不可打印字符也不能在URL中使用。另外，URL被设计时，只考虑到了ASCII码（历史原因)。但是，随着互联网的不断发展，从一开始只有美国使用互联网，到现在全球使用互联网，发展速度十分之快。这样，使得对URL的新需求也在不断增多，比如要求能够传输其他字符（西欧字符、罗马字符、汉字等等），但URL一开始只考虑到了ASCII码，并没有考虑其他编码（事实上，那时候还没有其他字符编码）。因此，为了满足新需求，设计者再次设计，最终通过如下方法解决这些问题:

 

1. 规定一些字符为特殊字符用于特殊目的

2. 使用转移符(%)使得能够传输西欧字符、汉字和不可打印符、空白符等等

 

 

1. URL转义符

为了避开不同协议字符集限制且满足传输各种字符，URL使用转义符(%)。这种转移表示法包含一个百分号(%),后面跟着两个标识字符ASCII码的16进制数来表示一个字节码，如下:

URN

URI的第二种形式就是统一资源名(URN)。URN作为特定内容的唯一名称使用的，与目前资源所在地无关。使用这种与位置无关的URN，就可以将资源四处搬移，哈可以用同一个名字通过多种网络访问协议来访问资源。

比如，不论因特网标准文档RFC 2141位于何处，都可以用下面的URN来命名它:

urn:ietf:rfc:2141

由于其实现困难，目前任处于实验阶段，还未正式使用
